# Sprint 迭代协议 (Sprint Protocol)

## 1. 迭代定义
项目按 **Sprint 1, 2, 3...** 分阶段推进。每个 Sprint 代表一个可交付的里程碑。

### 最佳实践：垂直切片 (Vertical Slices)
**强烈建议**按“功能特性”进行垂直切片开发。一个任务应包含从数据流到接口的完整路径。

## 2. 迭代生命周期 (基于文档迭代协议)

每个阶段的文档产出均需遵循 `01_WORKFLOW.md` 中的 **文档迭代协议**。

### 2.1 启动阶段 (Planning)
1.  **制定总计划**:
    *   创建 `docs/sprints/sprint_{xx}_plan.md`。
    *   通过 QA 迭代确认本次 Sprint 的需求范围、优先级和目标。
2.  **拆分具体子任务**:
    *   为复杂任务创建 `docs/sprints/sprint_{xx}_task_{yy}_{name}.md`。
    *   **深度细节**: 子任务文档必须细化到“实现规格”级别，消除所有逻辑盲点。

### 2.2 执行阶段 (Execution)
1.  按照子任务计划执行 **核心开发循环**。
2.  若开发中发现设计不合理，需更新对应的子任务文档或技术文档。

### 2.3 结项阶段 (Closing)
1.  **编写总结**: 产出 `docs/archive/sprint_{xx}/summary.md`。
2.  **状态核对**:
    *   哪些需求已实现？
    *   哪些需求宣称要实现但最终落空？（标记为 Roll-over 或 Dropped）。
3.  **归档**: 整理过程文档，更新全局 `requirement.md` 中的状态。
4.  **技术债记录**: 在总结中记录本次迭代产生的技术债。

## 3. 状态管理
*   **Pending**: 需求池中，未排期。
*   **In Sprint**: 已被选入当前迭代。
*   **Implemented**: 代码已编写完成。
*   **Verified**: 经过验证，正式达成。
*   **Archived**: 流程结束，文档已归档。
